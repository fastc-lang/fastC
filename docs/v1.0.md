# fastC v1.0 — Stable Surface and Stability Commitment

This document declares what fastC v1.0 promises to keep working,
what's a preview that may change shape, and how the project handles
breaking changes after launch.

It is the contract a user signs when they pin their `fastc.toml` to
the v1.0 release.

## What's stable at v1.0

Every item in this section will continue to compile, type-check, and
behave identically across the entire v1.x line. Bug fixes that produce
*the same observable behaviour as the documented intent* are exempt;
anything else is a breaking change and requires a v2.0 cycle.

### Language core

- Functions, parameters, return types — including the `void`,
  primitive, pointer (`ref` / `mref` / `raw` / `rawm` / `own`),
  slice, array, optional (`opt`), and result (`res`) shapes.
- Structs, enums (with payloads), `impl` blocks for inherent methods.
- Traits with method declarations, default methods, generic bounds
  (`[T: Bound1 + Bound2]`), `impl Trait for Type` blocks.
- Built-in traits `Eq`, `Ord`, `Copy`, `Drop`, `Hash`, `Clone` with
  per-primitive impls shipped in the prelude.
- Generics: monomorphized functions, monomorphized structs, recursive
  generic functions, multi-parameter generics, generic structs as
  fields of non-generic structs.
- Expressions: literals, arithmetic with overflow trap, comparisons,
  logical (`&&` / `||` / `!`), bitwise, calls, qualified calls
  (`mod::fn`), method calls (`x.method()`), `if` / `if let`,
  `switch` / `case` / `default`, `while`, `for`, `break`,
  `continue`, `return`, `defer`.
- Closures (capture-free) lifted to top-level `__lambda_N` functions
  and consumed via `fn(T) -> R` typedef-backed function pointers.
- Pointer operators: `addr` / `addrm` / `deref` / `at` for raw-pointer
  indexing inside `unsafe { ... }`.
- Module system: `mod`, `use ... :: ...` with Single / Multiple /
  Glob / Module forms; module-qualified name resolution.
- Trait method dispatch at mono time: `x.method()` on a generic
  receiver lowers to `T_method(&x)` for the concrete `T`.

### Safety model

- Power-of-10 rules at three levels: `--safety-level=relaxed`,
  `standard` (default), `critical` (full P10 enforcement). The
  *set of rules* under each level is stable; the diagnostic text
  is allowed to improve.
- Capability-typed I/O — `mod caps` mints `CapFsRead`, `CapFsWrite`,
  `CapNetConnect`, `CapNetListen`, `CapProcSpawn`, `CapTimeRead`,
  `CapRand`, `CapEnvRead` and the bundled `Caps` struct. **The
  fabrication check (sealed structs only constructible inside
  `mod caps`) and the `caps::init`-from-`main` rule are stable.**
- Contracts: `@requires(<expr>)` / `@ensures(<expr>)` on every
  function; `result` keyword refers to the about-to-return value in
  `@ensures`. **The runtime-trap semantics (fail-fast on
  contract violation) are stable; the static-discharge tier (proven
  obligations elide the trap) is also stable.**
- The fastC manifest is closed-schema TOML (`#[serde(deny_unknown_fields)]`).
  Unknown top-level keys are an error. The supply-chain claim
  ("no executable build scripts, ever") is structural.

### Compilation & tooling

- `fastc compile <file>` — produces C11. Output is deterministic
  (same input bytes produce byte-identical C output, modulo the
  documented `--O` flags which control the C compiler invocation).
- `fastc check <file>` — type-check without emitting C. Stable
  exit codes (0 = ok, non-zero = error). `--output-format=json`
  emits machine-readable status.
- `fastc build` / `fastc run` — manifest-driven project build.
  `--reproducible` flag (C2 in v1.0.x) normalizes embedded source
  paths to basenames so multi-file builds produce byte-identical
  C across working directories.
- `fastc new` / `fastc init` — project scaffolding. The generated
  layout (`src/`, `fastc.toml`, `Makefile`/`CMakeLists.txt`/`meson.build`,
  `README.md`, `AGENTS.md`, `.gitignore`) is the stable shape.
- `fastc explain <file>` — JSON document of every function's
  signature, capabilities, contracts, doc comments, plus the v1.3
  structured annotations (`mem` / `panics` / `purity` / `complexity`
  / `is_test`) and a top-level `modules` array carrying any
  `//!` module headers. Schema is stable (additive — new fields
  may appear, existing fields keep meaning).
- `fastc context <file>` — markdown (default) or JSON dump of the
  project's pub type surface, optimized for AI context windows.
  Signatures only, no bodies.
- `fastc diff <old> <new>` — semantic AST-level diff between two
  source snapshots. Added / removed / signature-changed pub items.
- `fastc fmt` — formats fastC source. Idempotent.
- `fastc cert-report` — Power-of-10 compliance evidence in
  JSON / text / compact.
- `fastc target list` / `fastc target check` — cross-compile
  matrix introspection.
- `fastc build --target=<triple>` for the eight v1.9 targets
  (aarch64/x86_64 × linux-musl/linux-gnu, aarch64/x86_64-macos,
  wasm32-wasi, riscv64-linux-musl).
- `fastc add <url>` / `fastc lock` — vendor-first dependency flow.
  `sha256` content-hash verification on every fetch.
- `fastc bench` — compile-time budget gate.
- `fastc mcp` — MCP stdio server exposing `explain`, `check`,
  `compile`, `caps_summary`, `context`, `diff` tools (B5 in v1.0.x).

### v1.3 annotation surface

Function-level structured annotations (A1 in v1.0.x):

- `@mem(arena = ident)` — names the memory region a fn allocates
  from. Documentation-only in v1.x; arena-aware allocators land
  in v2.x.
- `@panics(never | always | on = expr)` — declares the panic
  surface. `@panics(never)` is enforced against the transitive
  call graph (no path to `fc_trap` / `panic` / `abort` / `exit`).
  `always` and `on=expr` are documentation in v1.x.
- `@purity(pure | effect | io)` — observable-effect classification.
  `@purity(pure)` is enforced (no alloc, no I/O, no logging).
  `effect` and `io` are documentation.
- `@complexity(O(<shape>))` — informational complexity bound.
  Recognized shapes: `1`, `n`, `log n`, `n log n`, `n^k`, `2^n`.

Module-level mandatory headers (A2 in v1.0.x) — `//!` lines at the
top of an inline `mod foo { ... }` block:

```
mod alpha {
    //! @module = "alpha"
    //! @owns = "alpha"
    //! @arch = "core"
    //! @depends = "beta"
    //! @threading = "single"
    //! @invariants = "non-negative inputs"
    ...
}
```

Lenient mode (default for back-compat): header-less modules
parse fine; a module with any `//!` line must have the complete
required-key set. Cross-module checks (`@owns` uniqueness,
`@depends` exhaustiveness, `@arch` DAG layering) fire when any
module declares a header.

### Standard library (prelude modules)

The **public** surface of each shipped prelude module is stable:

- `mod math` — `abs_*`, `min[T: Ord]`, `max[T: Ord]`, `clamp[T: Ord]`.
- `mod mem` — `alloc(size)`, `free_bytes(ptr)`, `resize`.
- `mod io` — `println`, `put_char`, `print_int`.
- `mod log` — `debug`, `info`, `warn`, `error`, `kv_int`, `kv_str`.
- `mod json` — `JsonBuilder` plus `find_int` decoder.
- `mod toml` — `find_int`, `find_bool`.
- `mod cli` — `count`, `arg_at`, `program_name`, `has_flag`,
  `flag_value`, `flag_int`, `null_arg`, `is_null`.
- `mod http` — `get_status(c: ref(CapNetConnect), ...)`.
- `mod time` — `now(c: ref(CapTimeRead))`.
- `mod env` — `get(c: ref(CapEnvRead), key)`.
- `mod fs` — `exists(c: ref(CapFsRead), path)`,
  `size_bytes(c: ref(CapFsRead), path)`.
- `mod rand` — `seed(c, s)`, `next_u32(c)`.
- `mod caps` — `init() -> Caps` (callable only from `main`).
- `mod vec` — `Vec[T]`, full container surface
  (`new` / `with_capacity` / `push` / `pop` / `get` / `len` / `is_empty`
  / `clear` / `contains` / `release` / `map` / `filter` / `for_each`
  / `reduce` / `swap` / `reverse` / `extend` / `clone` / `sort` /
  `min_of` / `max_of` / `any` / `all`).
- `mod hashmap` — `HashMap[K: Hash + Eq, V]`, full container surface
  (`new_map` / `with_cap_map` / `put` / `lookup` / `drop_key` /
  `has_key` / `count_map` / `empty_map` / `release_map`).
- `mod str` — `Str` plus byte-level construction / inspection /
  search / formatting helpers.

### Supply chain

- Manifest dependency format:
  `name = { git = "...", rev = "...", sha256 = "...", sigstore = "..." }`.
- `fastc.lock` schema with the `sha256` field for content-hash
  verification.
- The `.github/workflows/release.yml` flow — cosign keyless +
  SLSA L3 — is the long-term release pipeline.
- Verification commands (`cosign verify-blob --bundle ...`,
  `slsa-verifier verify-artifact ...`) work against every v1.x release.

## What's a preview at v1.0

These features are shipped and working, but their *shape* may change
in a v1.x release. Each carries a `*(preview)*` marker in the docs.

- **Body-aware SMT contract discharge.** ✅ shipped in v1.0.x — the
  tier-2 encoder now feeds Z3 the function's `@requires` clauses as
  assumptions and a model for `result` derived from straight-line
  function bodies. Proves `@requires(x > 0) @ensures(result > 1)`
  for `return (x + 1);` and similar shapes.
- **Call-site precondition discharge.** ✅ shipped in v1.0.x — every
  direct call `f(args)` is re-checked against `f`'s `@requires`
  using argument substitution. Tier-1 catches literal-arg calls
  (`divisor_safe(5)` substitutes `x → 5`, folds to `5 > 0`); tier-2
  proves caller-context propagation (`@requires(y > 0) caller(y) {
  return divisor_safe(y); }` proves the call is safe under SMT).
  Method dispatch is covered. **Function-pointer bindings**
  (N1) — `let f = direct_fn; apply(f, x)` — are tracked in
  `callsite.rs` via a `fn_ptr_bindings` map threaded through
  the walker and resolved back to the underlying free fn at the
  call site, so the obligation discharges as if the user wrote
  `apply(direct_fn, x)` directly. Per-call-site obligations
  appear in `discharge.json` with `"clause": "call_site"`. Truly
  opaque fn-pointer *parameters* (the callee's closure source isn't
  visible) still fall through — whole-program callee inference is
  a v2.x slice.
- **fastc-core vendor packages.** ✅ split in v1.0.x — the five
  stage-1.8 launch-set modules (`mod cli` / `mod log` / `mod json` /
  `mod toml` / `mod http`) now exist as standalone public repos
  at
  [`Skelf-Research/fastc-core-{cli,log,json,toml,http}`](https://github.com/Skelf-Research)
  with their own `fastc.toml` / `README.md` / `AGENTS.md` / `LICENSE`
  and a `src/lib.fc` documentation stub of the final API. The
  public API ships inside the fastC prelude for v1.0 (no
  user-facing migration); cutting the prelude over to vendored
  `fastc add` consumption is a v1.1 packaging slice with no API
  change.
- **Global build cache** ✅ shipped in v1.0.x — `~/Library/Caches/fastc/build/`
  stores the emitted C keyed by `(source, fastc_version, safety_level,
  target_triple, header_flag, strict_flag)`. Cold build → warm build
  drops from ~640 ms to ~18 ms for the hello case (35× speedup) on
  M3. Skipped when `--prove`, `--no-prove`, header emission, or
  timing collection is in play (those need a fresh pipeline pass).
- **Discharge result on-disk cache** ✅ shipped in v1.0.x —
  `.fastc/cache/discharge/<formula-hash>.bin` per-project. Tier-2
  SMT calls drop from ~470 ms to ~26 ms on rerun (18× speedup).
- **Closures with by-value captures.** ✅ shipped in v1.0.x (N4) for
  the common case — closures referencing outer `let x = <literal>`
  bindings work via constant-inlining (IntLit / BoolLit / FloatLit
  + unary-negated literals). The desugar pipeline gained a
  `constant_scopes` stack on `ClosureLifter` plus an
  `inline_captures_in_block/stmt/expr` walker (with full
  `Stmt::If` / `While` / `Block` / `Unsafe` recursion) that
  substitutes captures into the closure body before lifting.
  Closure params are excluded from substitution via a HashSet.
  `crates/fastc/tests/closure_captures.rs` covers int literal,
  bool-in-if-condition, and negative-literal cases end-to-end
  through `cc`. Non-literal captures (function results, struct
  fields, mutable bindings) still emit the closure-aware
  "undefined name" diagnostic — full env-struct synthesis is the
  v2.0 follow-up.
- **Pipeline fuzzing.** ✅ shipped in v1.0.x — `crates/fastc/fuzz/`
  carries four libfuzzer targets covering parse, check (resolve +
  typecheck + cap_check + noalloc + p10), full compile (down to
  emit), and discharge. `.github/workflows/fuzz.yml` runs them as
  a matrix on every PR that touches the lexer, parser, or fuzz
  harness, with per-target time budgets tuned to per-iteration
  cost. A reproducing crash uploads as a CI artifact.
- **Cross-directory reproducibility.** ✅ shipped in v1.0.x —
  `fastc compile --reproducible` normalizes the source path
  embedded in `#line` directives to the basename so the same
  `.fc` produces byte-identical C output regardless of working
  directory. `crates/fastc/tests/reproducibility.rs` verifies the
  property end-to-end.
- **Compiler-binary reproducibility.** ✅ shipped in v1.0.x (N5) —
  `.github/workflows/release.yml`'s "Pin reproducibility env" step
  binds `SOURCE_DATE_EPOCH` to the tagged commit's author
  timestamp and sets
  `RUSTFLAGS=--remap-path-prefix $(pwd)=/fastc --remap-path-prefix $HOME/.cargo=/fastc-deps`,
  so every published `fastc-<os>-<arch>` binary is byte-identical
  when rebuilt from the same tag on any machine. `docs/supply-chain.md`
  carries the local-reproduction recipe.
- **`[workspace]` manifest + per-member incremental.** ✅ shipped in
  v1.0.x (N3) — `fastc.toml` accepts `[workspace] members = [...]`
  at the project root. `fastc build` at the workspace root iterates
  each member with its own M1 project cache; editing only
  `<member>/src/main.fc` re-emits that member and serves the others
  from cache. 40× warm-vs-cold on the two-member smoke test
  (526ms → 13ms). `--cc` / `--target` / `--cc-override` are
  rejected at the workspace root (they live on member builds).

## Breaking-change policy after v1.0

A change is **breaking** when any of these is true:

- Existing fastC source that compiles under v1.0 fails to compile
  (or produces different observable behaviour) under v1.x.
- A documented JSON artifact's schema removes a field or changes a
  field's meaning.
- A documented CLI subcommand or flag is removed or has its semantics
  changed (additive flags are fine).
- A prelude module's public function signature changes type.

The project's rule is:

1. **Breaking changes go in a v2.0 cycle.** A v2.0 release has its
   own roadmap, its own migration document, and a `fastc-migrate`
   tool that rewrites v1.x source to the v2.0 shape where possible.
2. **Bug fixes that align observable behaviour with documented
   intent are not breaking,** even if existing source relied on the
   bug. The fix is noted in the changelog and ships in v1.x.
3. **Additive changes are always allowed in v1.x.** New CLI flags,
   new prelude modules, new annotation kinds, new target triples
   — none of these are breaking.
4. **Compiler internals are not stable.** The AST, the lower-pass
   IR, the C output formatting, the on-disk cache layout, the MCP
   tool descriptors — all of these can change without notice.

## How to depend on fastC v1.0

In your project's `fastc.toml`:

```toml
[package]
name = "your-app"
version = "0.1.0"
fastc-version = "1.0"
```

(The `fastc-version` field is a v1.0-introduced manifest key. fastC
itself uses it to refuse compilation if your declared compiler version
is older than what your source actually needs.)

In CI:

```sh
# pin to an exact release for reproducible builds
curl -sSL https://github.com/Skelf-Research/fastc/releases/download/v1.0.0/fastc-linux-x86_64 \
  -o fastc
cosign verify-blob \
  --bundle fastc-linux-x86_64.sigstore.json \
  --new-bundle-format \
  --certificate-identity-regexp '^https://github.com/Skelf-Research/fastc/\.github/workflows/release\.yml@.+$' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  fastc
chmod +x fastc
```

## What v1.0 is and isn't

**v1.0 is** the smallest set of features that lets you build a
real, deployable cap-typed application end-to-end:

- Argv parsing, structured logging, config parsing, HTTP fetch,
  JSON field extraction — all via the launch-set modules.
- Cross-compilation to 8 targets out of the box.
- Verified supply chain — `sha256` + Sigstore + SLSA L3 on the
  compiler binary; per-dep content hash on every fetch.
- Contract discharge that proves some obligations statically and
  preserves the runtime safety net for everything else.
- Power-of-10 enforcement at the level you choose.
- Agent-facing artifacts (`fastc explain` JSON, `caps.json`,
  `discharge.json`, `AGENTS.md`).

**v1.0 isn't:**

- A general-purpose stdlib replacement for C / Rust / Go's
  ecosystems. fastc-core ships five packages, not five hundred.
- A drop-in for legacy C codebases. fastC is C11-output-compatible,
  not C-input-compatible. `@cImport`-style header consumption is
  not in scope.
- A WebAssembly-first runtime. `wasm32-wasi` is one of eight
  cross-compile targets, not the primary deployment story.
- An async-first language. `async`/`await` is roadmap item 2.3
  and may not land in v1.x at all.
